title: Secrets
description: Managing encrypted secrets with agenix across all hosts.
---

Secrets are managed with [agenix](https://github.com/ryantm/agenix), a tool that encrypts files with [age](https://age-encryption.org) and decrypts them at activation time using the host's SSH host key (or a user's SSH key). Each secret is an `.age` file committed to the repository; the plaintext is only ever accessible on machines whose public keys were listed as recipients when the file was encrypted.

## How it works

Every `.age` file in `secrets/` has a list of recipient public keys defined in `secrets/secrets.nix`. When agenix decrypts a secret during `nixos-rebuild`, it uses the host's `/etc/ssh/ssh_host_ed25519_key` (or the user key `~/.ssh/id_ed25519`) — whichever was listed as a recipient.

At runtime, decrypted secrets land in `/run/agenix/` and are referenced via `config.age.secrets.<name>.path`.

## The agenix module

The NixOS module imports the upstream agenix NixOS module and adds the `agenix` CLI tool to `systemPackages` so you can re-key and create secrets from any host:

```nix
# modules/nixos/agenix.nix
{ inputs, perSystem, ... }:
{
  imports = [ inputs.agenix.nixosModules.default ];
  environment.systemPackages = [ perSystem.agenix.default ];
}
```

This module is included in every host through the `base` profile (`modules/nixos-profiles/base.nix`).

## Secret inventory

`secrets/secrets.nix` defines which public keys may decrypt each secret. Four recipient groups are used:

| Group | Keys included |
|---|---|
| `baremetal` | `user-jel`, `host-dreamweaver`, `host-hlebak`, `host-hydromechanizator` |
| `server` | `user-jel`, `host-dreamweaver` |
| `userspace` | `user-jel`, `host-hlebak`, `host-hydromechanizator` |
| `wsl` | `user-jel`, `host-wsl` |

```nix
# secrets/secrets.nix
let
  user-jel = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIIeyqBh/je0A8dMXB0m9jxZ0nBWHc5Hj+dFmFBiT5Mdw";
  host-dreamweaver = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGyWXKJDJlphwmyqE8O75EKf8/xbUYrminvff3NZu7/k";
  host-hlebak = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIOLi8mbXX3+fmtHj47rM0wWcwcfgivmuXeB2oSIA/1yf";
  host-hydromechanizator = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIKpc44dkGAK435at+4wQbrgfv21hqXe03rQrna93QT3u";
  host-wsl = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHu2STpuDoldavVByGvUN9e+tP/BfTorGkiixYGVO/4G";

  baremetal = [ user-jel host-dreamweaver host-hlebak host-hydromechanizator ];
  server    = [ user-jel host-dreamweaver ];
  userspace = [ user-jel host-hlebak host-hydromechanizator ];
  wsl       = [ user-jel host-wsl ];
in
{
  "cloudflared-certificate.age".publicKeys          = server;
  "cloudflared-tunnel-dreamweaver.age".publicKeys   = server;
  "copyparty-admin.age".publicKeys                  = server;
  "freenode-je-password.age".publicKeys             = userspace;
  "jel-password.age".publicKeys                     = baremetal;
  "wireless-networks.age".publicKeys                = baremetal;
  "wsl-password.age".publicKeys                     = wsl;
}
```

| Secret file | Recipient group | Purpose |
|---|---|---|
| `cloudflared-certificate.age` | `server` | Cloudflare origin certificate |
| `cloudflared-tunnel-dreamweaver.age` | `server` | Cloudflare tunnel credentials |
| `copyparty-admin.age` | `server` | copyparty admin password |
| `freenode-je-password.age` | `userspace` | IRC password |
| `jel-password.age` | `baremetal` | User account password hash |
| `wireless-networks.age` | `baremetal` | WPA/NetworkManager config |
| `wsl-password.age` | `wsl` | WSL user password hash |

## Referencing secrets in modules

Modules declare which `.age` files they need and then read the runtime path from `config.age.secrets.<name>.path`. The cloudflared module is a representative example:

```nix
# modules/nixos/cloudflared.nix
{ config, ... }:
{
  age.secrets = {
    cloudflared-certificate.file = ../../secrets/cloudflared-certificate.age;
    cloudflared-tunnel-dreamweaver.file = ../../secrets/cloudflared-tunnel-dreamweaver.age;
  };

  services.cloudflared = {
    enable = true;
    certificateFile = config.age.secrets.cloudflared-certificate.path;

    tunnels."73bc3393-f815-4a60-8f3a-c9d8f6d2e0dc" = {
      credentialsFile = config.age.secrets.cloudflared-tunnel-dreamweaver.path;
      default = "http_status:404";
    };
  };
}
```

The pattern is:
- `age.secrets.<name>.file` — path to the `.age` file in the repo (resolved at build time)
- `config.age.secrets.<name>.path` — path to the decrypted file in `/run/agenix/` (resolved at runtime)

## Adding a new secret

<Steps>
  <Step title="Create the encrypted file">
    From a machine whose key is in `secrets/secrets.nix`, run `agenix` to create the new file. The `-e` flag opens your `$EDITOR` and encrypts on save:

    ```bash
    cd secrets
    agenix -e my-new-secret.age
    ```

    If the secret does not yet exist in `secrets.nix`, agenix will warn that no recipients are configured. Add the entry first (next step), then come back and run this.
  </Step>
  <Step title="Declare recipients in secrets.nix">
    Open `secrets/secrets.nix` and add an entry for the new file, choosing the appropriate recipient group:

    ```nix
    "my-new-secret.age".publicKeys = server; # or baremetal, userspace, wsl
    ```
  </Step>
  <Step title="Re-encrypt with the correct recipients">
    After updating `secrets.nix`, re-key the file so it is encrypted for all declared recipients:

    ```bash
    agenix -r -i ~/.ssh/id_ed25519
    ```
  </Step>
  <Step title="Reference in a NixOS module">
    In the module that needs the secret, declare the file path and use `config.age.secrets.<name>.path` wherever the plaintext is required:

    ```nix
    { config, ... }:
    {
      age.secrets.my-new-secret.file = ../../secrets/my-new-secret.age;

      services.myservice.passwordFile = config.age.secrets.my-new-secret.path;
    }
    ```
  </Step>
  <Step title="Commit the .age file">
    Commit the new `.age` file and the updated `secrets.nix`. The encrypted ciphertext is safe to store in the repository.

    ```bash
    git add secrets/my-new-secret.age secrets/secrets.nix
    git commit -m "secrets: add my-new-secret"
    ```
  </Step>
</Steps>

## Re-keying after SSH key changes

When a host is reinstalled or its SSH host key changes, existing secrets encrypted for that host can no longer be decrypted by it.

<Warning>
After replacing or rotating an SSH host key, you must re-key **all** secrets that list the affected host as a recipient. Until you do, any service that reads a secret on that host will fail to start.
</Warning>

Update `secrets/secrets.nix` with the new public key, then re-encrypt every affected file from a machine that already has decryption access:

```bash
# Re-key all secrets at once
agenix -r -i ~/.ssh/id_ed25519
```

Commit the re-encrypted `.age` files and rebuild the host.

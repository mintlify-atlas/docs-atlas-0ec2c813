`nix-configuration` is a personal NixOS flake that fully manages every machine its owner uses — from a desktop workstation and a laptop to a home server and Windows Subsystem for Linux environments. Every piece of system state, every installed package, every daemon, and every secret is tracked in version control and rebuilt from source.

<Note>
  This is a personal configuration, not a general-purpose framework. The documentation describes how it works so that the design decisions, module structure, and host layouts are easy to navigate and understand.
</Note>

## What it is

At its core, `nix-configuration` is a single [Nix Flake](https://nixos.wiki/wiki/Flakes) that declares the desired state of several machines. When you run `nixos-rebuild switch --flake .#hostname`, NixOS computes the entire system closure — every package, configuration file, and service — from the flake and activates it atomically. Rolling back is a single command.

The configuration is built on two complementary tools:

- **NixOS** manages the operating system: kernel, services, users, networking, storage, secrets, and hardware quirks.
- **Home Manager** manages user-space: dotfiles, application settings, shell configuration, and per-user packages.

Both layers share the same flake, the same module system, and the same set of reusable profiles — so changes made in one place are reflected everywhere that imports the relevant module.

## Key technologies

  <Card title="NixOS" icon="linux" href="https://nixos.org">

  <Card title="Nix Flakes" icon="snowflake" href="https://nixos.wiki/wiki/Flakes">

  <Card title="Home Manager" icon="house" href="https://github.com/nix-community/home-manager">

  <Card title="blueprint" icon="map" href="https://github.com/numtide/blueprint">

  <Card title="Stylix" icon="palette" href="https://github.com/nix-community/stylix">

  <Card title="agenix" icon="lock" href="https://github.com/ryantm/agenix">

### Additional inputs

Beyond the six above, the flake also pulls in:

- **nixos-hardware** — hardware-specific NixOS modules (used for the Framework 16 laptop).
- **NixOS-WSL** — NixOS support for Windows Subsystem for Linux.
- **srvos** — opinionated NixOS modules for servers and workstations from numtide.
- **nix-vscode-extensions** — a large, frequently updated set of VS Code extension packages.
- **flake-firefox-nightly** — nightly Firefox builds packaged for Nix.
- **copyparty** — a self-hostable file server, included as a flake input so its version is pinned.
- **catppuccin-fish** — the Catppuccin theme for Fish shell (included as a plain source archive, `flake = false`).

## Design philosophy

The configuration is built around three principles:

**Declarative.** You describe *what* the system should look like, not *how* to get there. Adding a service means adding a module import — not running an imperative install script. Removing it means removing the import and rebuilding.

**Composable.** Functionality is split into small, focused modules (one file per service or tool) and then grouped into profiles. A host picks the profiles appropriate for its role. The `base` profile runs on every machine; `userspace` adds a graphical environment; `server` adds self-hosted services. Nothing is duplicated across hosts.

**Reproducible.** Every input is pinned in `flake.lock`. Two machines importing the same profile with the same locked inputs produce bit-for-bit identical derivations. Upgrading a dependency is a deliberate, reviewable change to `flake.lock`.

<Info>
  The combination of blueprint's auto-discovery and small single-purpose modules keeps `flake.nix` to 59 lines — the entire output expression is `outputs = inputs: inputs.blueprint { inherit inputs; };`.
</Info>

The entire repository is a single Nix Flake. Everything тАФ hosts, modules, secrets, assets тАФ lives under one root, and the outputs are derived automatically from the directory layout by the `blueprint` framework. This page explains how all the pieces connect.

## The flake.nix file

`flake.nix` is intentionally minimal. All 59 lines are inputs and a one-line output expression:

```nix flake.nix
{
  inputs = {
    agenix = {
      url = "github:ryantm/agenix";
      inputs.nixpkgs.follows = "nixpkgs";
    };

    blueprint = {
      url = "github:numtide/blueprint";
      inputs.nixpkgs.follows = "nixpkgs";
    };

    # ... (remaining inputs) ...

    nixpkgs.url = "github:NixOS/nixpkgs/nixos-25.11";
  };

  outputs = inputs: inputs.blueprint { inherit inputs; };
}
```

The `outputs` line delegates everything to `blueprint`. There is no manual `nixosConfigurations`, `homeConfigurations`, or `packages` attribute set тАФ blueprint generates them by scanning the repository.

## How blueprint works

[blueprint](https://github.com/numtide/blueprint) is a flake framework that turns a structured directory layout into flake outputs. The entire output expression is:

```nix flake.nix
outputs = inputs: inputs.blueprint { inherit inputs; };
```

When you pass the flake's `inputs` to `blueprint`, it:

1. Scans `hosts/` for directories containing a `configuration.nix`, and creates a `nixosConfigurations` entry for each one.
2. Scans `modules/nixos/` for `.nix` files and exposes them as `nixosModules`.
3. Scans `modules/home/` for `.nix` files and exposes them as `homeModules`.
4. Makes any additional directories (like `modules/nixos-profiles/` and `modules/home-profiles/`) available under `flake.modules`.

Inside every module and host configuration, `flake` is passed as a special argument. This gives every file access to `flake.nixosModules`, `flake.homeModules`, and `flake.modules.nixos-profiles` / `flake.modules.home-profiles` without any manual wiring.

<Note>
  Because blueprint discovers modules by filename, adding a new NixOS module is as simple as creating a new `.nix` file in `modules/nixos/`. It appears in `flake.nixosModules` automatically.
</Note>

```

тФЬтФАтФА flake.nix                   # Inputs and single-line outputs
тФЬтФАтФА flake.lock                  # Pinned dependency revisions
тФЬтФАтФА hosts/                      # One directory per machine

тФВ   тФЬтФАтФА hlebak/
тФВ   тФЬтФАтФА hydromechanizator/
тФВ   тФЬтФАтФА hlebak-wsl/
тФВ   тФФтФАтФА spycharkozrywarka-wsl/
тФЬтФАтФА modules/
тФВ   тФЬтФАтФА nixos/                  # Individual NixOS modules (~70 files)
тФВ   тФЬтФАтФА home/                   # Individual Home Manager modules (~35 files)
тФВ   тФЬтФАтФА nixos-profiles/         # Bundles of NixOS modules by role

тФЬтФАтФА symbols/                    # Icon/symbol assets
тФФтФАтФА wallpapers/                 # Wallpaper images used by Stylix
```

### `hosts/`

Each subdirectory corresponds to one machine. A host directory contains at minimum a `configuration.nix` that imports profiles and sets host-specific values (kernel modules, `hostId`, `hostPlatform`, `stateVersion`), and a `users/` directory with Home Manager configurations for each user on that host.

### `modules/nixos/`

Contains ~70 single-purpose NixOS module files. Each file configures one service or feature тАФ `docker.nix`, `gnome.nix`, `tailscale.nix`, and so on. These are exposed as `flake.nixosModules` and are consumed almost exclusively through profiles, not imported directly by hosts.

### `modules/home/`

Contains ~35 single-purpose Home Manager module files тАФ one per application or tool. Examples: `fish.nix`, `helix.nix`, `vscode.nix`, `ghostty.nix`. Exposed as `flake.homeModules`.

### `modules/nixos-profiles/` and `modules/home-profiles/`

Profiles are bundles: each profile file imports a set of related modules for a particular machine role. See [The profile system](#the-profile-system) below.

### `secrets/`

Contains age-encrypted secret files (`.age`) and a `secrets.nix` manifest that maps each secret to the set of public keys that can decrypt it.

### `assets/`, `symbols/`, `wallpapers/`

Static files used by the configuration тАФ icon assets, symbols, and wallpaper images referenced by the Stylix theming module.

## The profile system

Profiles solve a layering problem: most modules should be active on multiple machines, but importing 70 individual modules in every host `configuration.nix` would be unmanageable. Profiles group related modules into named bundles that hosts import by role.

## How a host configuration comes together

A host `configuration.nix` picks its profiles and then adds the minimum amount of host-specific configuration (kernel modules for hardware detection, a ZFS `hostId`, and `stateVersion`):

<Steps>
  <Step title="Import profiles">
    The host selects which profiles match its role. For example, `hlebak` is a physical desktop workstation with a GPU:

    ```nix hosts/hlebak/configuration.nix
    { flake, ... }:
    {
      imports = with flake.modules.nixos-profiles; [
        baremetal
        base
        nvidia
        userspace
      ];

      boot = {
        initrd.availableKernelModules = [
          "ahci"
          "nvme"
          "sd_mod"
          "usb_storage"
          "usbhid"
          "xhci_pci"
        ];

        kernelModules = [ "kvm-intel" ];
      };

      networking.hostId = "61e6a400";
      nixpkgs.hostPlatform = "x86_64-linux";
      system.stateVersion = "25.11";
    }
    ```
  </Step>
  <Step title="Profiles expand to modules">
    Each imported profile expands to its list of `nixosModules`. `base` brings in `agenix`, `doas`, `fish`, `nix`, `openssh`, and the rest. `baremetal` brings in `boot`, `zfs`, `stylix`, `tailscale`, and so on.
  </Step>
  <Step title="Modules are evaluated together">
    NixOS merges all imported modules тАФ from profiles, from direct imports, and from host-specific settings тАФ into a single system configuration. Modules declare options; NixOS resolves conflicts and builds the final system.
  </Step>
  <Step title="blueprint exposes the result">
    blueprint registers the evaluated configuration under `nixosConfigurations.hlebak` in the flake outputs, making it available to `nixos-rebuild switch --flake .#hlebak`.
  </Step>
</Steps>

## The module system

<CardGroup cols={2}>
  <Card title="nixosModules" icon="gear">
    Files in `modules/nixos/` become `flake.nixosModules`. Each is a standard NixOS module тАФ it can declare options, set config values, and reference other modules. They are only active when imported.
  </Card>
  <Card title="homeModules" icon="house">
    Files in `modules/home/` become `flake.homeModules`. Each is a standard Home Manager module scoped to one user. Used in `users/` directories within each host.
  </Card>
</CardGroup>

Both module sets use the same `flake` special argument pattern. A module that needs to reference another module (for example, to set an option defined by a sibling module) can access it through `flake.nixosModules` or `flake.homeModules`.

<Info>
  Because every module lives in its own file and is imported only when needed, it is straightforward to add, remove, or audit any single feature. Searching for `flatpak` in the repository immediately shows the one module file that enables it and the one profile that imports it.
</Info>

## Secrets layout

The `secrets/` directory pairs encrypted files with a declaration of who can read them:

```nix secrets/secrets.nix
let
  user-jel = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIIeyqBh/...";
  host-dreamweaver = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGyWXKJ...";
  host-hlebak = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIOLi8mbX...";
  host-hydromechanizator = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIKpc44dk...";
  host-wsl = "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHu2STpu...";

  baremetal = [ user-jel host-dreamweaver host-hlebak host-hydromechanizator ];
  server    = [ user-jel host-dreamweaver ];
  userspace = [ user-jel host-hlebak host-hydromechanizator ];
  wsl       = [ user-jel host-wsl ];
in
{
  "cloudflared-certificate.age".publicKeys = server;
  "cloudflared-tunnel-dreamweaver.age".publicKeys = server;
  "copyparty-admin.age".publicKeys = server;
  "freenode-je-password.age".publicKeys = userspace;
  "jel-password.age".publicKeys = baremetal;
  "wireless-networks.age".publicKeys = baremetal;
  "wsl-password.age".publicKeys = wsl;
}
```

Each secret is encrypted to a group of public keys. Only the listed hosts (and the user key) can decrypt the secret at activation time. See [Secrets](/operations/secrets) for the full workflow.
